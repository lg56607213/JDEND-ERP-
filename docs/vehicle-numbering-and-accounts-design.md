# 차량 취득 회계·번호체계 설계

> 결정일: 2026-07-11 (총괄 사장 결정). 이 문서가 구현 기준(계약)이다.

## 확정된 결정

1. **기타계정관리 = 재무제표관리 연동 (하드코딩 제거).**
   - 기타계정관리(총괄관리>회계계정관리)의 계정 드롭다운은 **재무제표관리에 등록된 계정만** 뜨고 선택 가능.
   - 재무제표관리(`/api/financial-statement-accounts`, 특히 `/leaves` 말단계정)가 **계정 마스터(단일 진실)**. 계정코드·계정명은 여기서만 온다.
   - 저장 시 선택한 계정코드가 재무제표관리에 존재하는지 **백엔드에서도 검증**(없으면 거부).

2. **선급렌트자산 미사용 (현 정책).**
   - 주 고객이 소규모(차량 먼저 구매)라, **선급 단계에서 '차량운반구'로 직접 등록**.
   - 선급렌트자산 방식(계약 먼저 → 차량 나중, 대형 장기렌트업체용)은 **추후 옵션**으로 남김.

3. **번호체계 (신규 생성분부터 적용, 기존 데이터 마이그레이션 안 함).**
   - 발주번호 = `J + YYMMDD + 순번3` → 예 `J260701001`
   - 차량관리번호 = `발주번호 + 순번3` → 예 `J260701001001`
   - **1발주 = 여러 대 가능**: 끝 3자리는 그 발주 안의 차량 순번(001, 002, …)
   - 차량관리번호는 **실행 시점에 채번(확정)**.

4. **차량관리번호 = 모든 차량 관련 전표의 공용 키(주민번호 격).**
   - 실행 후 감가상각·정비·보험·매각은 물론 **수납 전표도 차량관리번호로 기재**.

5. **실행 이관.**
   - 발주(선급) 때 생긴 전표를 실행 시 **차량관리번호로 이관**.
   - 선급을 차량운반구로 직접 잡으므로 **계정 대체는 불필요**, 전표 키를 **발주번호 → 차량관리번호로 갱신**.
   - (선급렌트자산을 쓰는 경우엔: 선급렌트자산→차량운반구 대체 + 세금과공과 등 부대비용도 차량관리번호로 이관 → 차대변 정합. 지금은 미사용.)

## 구현 단계

### Phase 1 — 지금 구현 (번호체계와 독립)
- 기타계정관리 프론트: 드롭다운을 재무제표관리 계정목록 API에서 **동적 로드**. 저장 시 계정코드 존재 검증(프론트+백엔드).
- 선급(`VehicleAdvanceService`): 기본 차변 = 차량운반구. 기타계정관리 매핑이 설정돼 있으면 **그 설정대로**(현재 항목명이 안 맞아 매번 기본값으로 새는 문제 수정).

> **구현 완료 (2026-07-11)**
> - `account_management.html`: 하드코딩 `<option>` 전부 제거 → 페이지 로드 시 `GET /api/financial-statement-accounts/leaves`로 계정목록 동적 로드. 선급 항목(prepaid-item)도 JS(`PREPAID_ITEMS`)로 렌더. 은행계좌 select은 `.bank-select`로 분리해 계정목록 주입에서 제외.
> - 저장 검증: `OtherAccountSettingsService.save` → `validateAccountCodesExist`가 settings 내 모든 `account` 코드를 `FinancialStatementAccountRepository.existsByAccountCode`로 검증, 미등록 코드는 저장 거부.
> - **항목명 불일치 원인 확인·수정**: 기타계정관리는 `인지세`, 선급화면(`vehicle_advance.html`)은 `인지대`로 달랐고 `외주장착비용` 항목이 기타계정관리에 없었음 → `PREPAID_ITEMS`를 선급화면 항목명과 정확히 일치시켜 매칭 복구. 미지정 항목은 `VehicleAdvanceService`가 기본값 `차량운반구`로 처리(현 정책, 선급렌트자산 미사용).
> - 참고: 기존 하드코딩 코드(2010 선급렌트자산·1110 차량운반구 등)는 실제 계정 마스터와 불일치했음. 마스터 실제 코드는 `100401 차량운반구`, `500211 세금과공과` 등이며 `선급렌트자산` 계정은 마스터에 없음.

### Phase 2 — 상세 설계 (2026-07-11 조사 반영)

#### 현재 구조 (조사 결과)
- `VehicleOrder` = 차량 1대. 식별자 **`vehicleMgmtNo` 하나뿐**(unique, length 20). 포맷 `J0001`(J+4자리), **발주 시점 채번**(`VehicleOrderService.generateNextMgmtNo`, :616). 별도 발주번호 없음 → **1발주=1차량**.
- `vehicleNo`(length 30) = 차량번호(번호판), 나중 입력.
- 하위 도메인(계약/선급/정비/보험/매각 등)은 대부분 **`vehicleMgmtNo`로 참조**(findByVehicleMgmtNo).
- **전표(`Voucher`)만 예외**: 차량을 `vehicleNo`(번호판)+`contractNumber`로만 참조(`Voucher.java:29-33`). `vehicleMgmtNo` 컬럼 없음 → 선급 등 초기 전표는 번호판 미발급 상태라 차량 연결이 약함.

#### 목표 대비 격차
1. 포맷: `J0001` → 발주번호 `J260701001` / 차량관리번호 `J260701001001`
2. 1발주:1차량 → **1발주:N차량**
3. 차량관리번호 채번 시점: 발주 시 → **실행 시**
4. 전표 공용키: 번호판 → **차량관리번호**

#### 설계 (제안)
**핵심**: 하위 도메인은 이미 차량관리번호로 참조 → 공용키 통일은 **`Voucher`에 `vehicle_mgmt_no` 컬럼 추가**하고 차량 관련 전표 생성 시 채우면 됨(`vehicleNo`는 표시용 유지).

**식별자 생애주기 (신규분):**
- **발주 시**: 발주번호 채번 `J+YYMMDD+순번3`. 차량 N대면 N개 차량행 생성, 각 행은 발주번호 공유 + 발주 내 순번(001,002…). 이 단계 전표(선급)는 **발주번호로 태그**.
- **실행 시**: 각 차량행에 차량관리번호 `= 발주번호+순번3` 확정. 그 차량의 발주단계 전표를 **차량관리번호로 이관(갱신)**.
- **실행 후**: 감가상각/정비/보험/매각/수납 등 모든 전표를 차량관리번호로 기재.

**변경 지점:**
- `VehicleOrder`: `order_no`(발주번호) 필드 추가 + 채번 신규 포맷. `vehicleMgmtNo` 확정을 실행 시점으로 이동.
- `Voucher`: `vehicle_mgmt_no` 컬럼 추가. 차량 관련 전표 생성부(선급/실행/감가상각/정비/보험/매각/수납)에서 채움.
- 실행(execute): 차량관리번호 확정 + 발주단계 전표 이관.
- 채번 유틸: 발주번호(J+YYMMDD+순번3), 차량관리번호(발주번호+순번3).

**신규분부터 적용**(기존 `J0001` 데이터 유지). 문자열이라 두 포맷 공존 무방.

#### 확정된 결정 (2026-07-11)
1. **1발주 N대 입력**: 발주 화면에서 **"대수" 입력 → N개 차량행 자동생성**(각 행 = 발주번호 공유 + 순번).
2. **채번 시점**: **발주 시 바로 차량관리번호(발주번호+순번3) 채번 + 실행 시 '취득 확정'**. (실행 시 새로 채번하지 않음)

→ 이 결정으로 설계 단순화: 발주 시점부터 차량관리번호가 존재하므로 **선급 전표도 처음부터 차량관리번호로 태그** 가능. "발주번호로 태그 후 실행 시 이관" 단계 불필요. 발주번호는 그룹 헤더(집계용).

#### 구현 sub-step (제안)
- **S1. 번호체계·발주 N대**: `VehicleOrder`에 `order_no`(발주번호) 필드 추가. 채번 유틸(발주번호 `J+YYMMDD+순번3`, 차량관리번호 `발주번호+순번3`). 발주 생성 시 대수 N 입력 → N개 차량행 즉시 생성·채번. (기존 J0001 데이터는 유지, 신규분부터 신규 포맷)
- **S2. 전표 공용키**: `Voucher`에 `vehicle_mgmt_no` 컬럼 추가. 차량 관련 전표 생성부(선급/실행/감가상각/정비/보험/매각/수납)에서 차량관리번호 기재.
- **S3. 실행 확정**: 실행 시 상태를 '취득 확정'으로. (선급 직접 차량운반구 정책이므로 계정 대체 없음)
- **주의(스키마)**: `vehicle_orders`·`vouchers`에 컬럼 추가 → 멀티테넌트 각 회사 DB + 신규회사 생성 SQL/템플릿에 반영 필요.

### Phase 2 확정 v2 (2026-07-11 상세)

**번호 포맷 확정:**
- 발주등록 시 = **10자리** `J + YYMMDD + 발주순번3` (예 `J260706001` = 26/07/06 1번 발주). 날짜별 순번(다음날 001 리셋).
- 실행 시 = **13자리** `10자리 + 대별순번3` (예: 2번 발주 5대 → `J260706002001`~`J260706002005`).
- 발주~선급 전표 = **10자리**로 기재. 실행 이후 전표 = **13자리**. 전표는 **항상 차량관리번호 보유**.

**기본 차량관리번호:** 차량 무관 전표(법인세·대표자 가수금 등)는 공용 기본값 **`J0000000000000`**(13자리) 사용. 마스터로 1건 생성. 차량별 목록·집계에서는 제외.

**모든 화면 기준값 = 차량관리번호** (계약번호 아님):
- 계약 조회는 **차량관리번호 / 계약번호 / 차량번호** 모두 가능.
- **돋보기 검색 팝업**(차량번호 → 차량관리번호)을 여러 화면 공용 컴포넌트로.

**추가 고려사항 (총괄 제기):**
1. **N대 발주 취득원가 대별 배분** — 선급(10자리)에 잡힌 차량운반구를 실행 시 대별(13자리)로 나눌지(감가상각·매각이 대별이라 필요). 1대 발주는 자명, N대만 규칙 필요.
2. 기본 차량관리번호는 차량 목록/조회/감가상각 대상에서 **제외** 처리.
3. 돋보기 검색은 기존 차량검색 API(VehicleSearch) 재사용, 공용 팝업으로.
4. 한 차량 : 여러 계약(재렌트) 가능성 — 차량관리번호↔계약번호 관계 확인.
5. 발주번호 순번 **동시성**(같은 날 동시 발주) 유니크 처리.
6. 전표엔 `vehicleNo`(번호판, 표시용)와 `vehicle_mgmt_no`(공용키) 병행.

**DB/스키마 반영 방법 (테스트 단계, 실데이터 없음):**
- 테넌트 스키마 = **템플릿 DB(`erp`)를 `CREATE TABLE ... LIKE` 복제**(TenantDatabaseService) → **템플릿 DB가 스키마 마스터**.
- 신규 컬럼 반영 절차:
  1. JPA 엔티티에 컬럼 추가(`VehicleOrder.orderNo`, `Voucher.vehicleMgmtNo`)
  2. **템플릿 DB(`erp`)에 컬럼 추가**(ALTER, 또는 로컬 `ddl-auto:update`) → 이후 생성 회사 DB는 자동 상속
  3. 기존 회사 DB(test 데이터뿐) → **drop 후 재생성이 가장 깔끔**(또는 ALTER)
  4. 배포용은 `src/main/resources/sql/`의 기존 마이그레이션 SQL 관례대로 스크립트 1개 추가

### Phase 2 확정 v3 — 선급/발주 워크플로우 (2026-07-12)

- **선급은 차량 1대씩 개별 입력** (균등배분 아님). 탁송료 등으로 대별 취득원가가 다를 수 있음.
- **N대 발주 표시**: 발주 화면에 차량관리번호(10자리 발주번호)가 **N개 동일하게** 표시됨. 각 행은 **제조사계약번호(makerContractNo)가 달라** 사용자가 하나씩 들어가 입력·구분.
- **선급 태깅 → 실행 시 대체**: 선급금액은 **발주번호(10자리)로 등록** → 그 차량 **실행 시** 그 차량에 등록된 선급이 **차량관리번호(13자리)로 대체(이관)**. (어느 차량인지는 제조사계약번호/차량행으로 특정)
- **UX 요구**:
  - 발주 화면의 **"출고완료변경(실행)" 버튼 옆에 "선급등록" 버튼** 추가 → 누르면 바로 선급 등록 화면 진입(현재는 선급화면에서 차량관리번호를 조회·선택해야 함).
  - **선급 화면에 `제조사계약번호` 필드 추가** — 차량관리번호(13자리) 확정 전에 선급을 등록하는 경우가 대다수이므로, 차량번호와 함께 구분용으로.
- **구현 유의**: 발주 단계엔 N행이 같은 10자리를 공유 → 내부적으로 각 차량행/제조사계약번호로 특정, 실행 시 13자리 확정 + 그 차량 선급 전표를 10→13으로 재태그. (`vehicle_orders.vehicle_mgmt_no` 유니크 제약을 발주단계 공유가 가능하도록 조정 필요.)

### Phase 2 확정 v4 — 재렌트 번호체계 (2026-07-12)

13자리 차량관리번호의 **끝 3자리 = [재렌트회차 1자리][대순번 2자리]**.
- `J260712001` = J + YYMMDD(260712) + 발주순번3(001) → **10자리 발주번호**
- `J260712001001` = 최초계약(재렌트0), 01대
- `J260712001101` = 1회 재렌트, 01대
- `J260712001201` = 2회 재렌트, 01대
- (5대 발주 최초계약: `...001`~`...005` = 재렌트0 + 대01~05)

**의미 / "한 차량 : 여러 계약" 해결:**
- **물리적 차량(대)** = 발주번호 + 대순번(끝 2자리). 재렌트해도 동일.
- **계약 인스턴스** = 전체 13자리(재렌트회차 포함). 재렌트마다 새 번호.
- → 각 계약(재렌트 포함)이 차량관리번호로 구분되므로, 수납/청구를 차량관리번호로 하면 어느 계약인지 자동 특정.

**한계:** 재렌트 최대 9회(1자리), 발주당 대수 최대 99대(2자리). 실무상 충분, 초과 시 자릿수 규칙 확장.

**중요 구분 (S2 처리):**
- **자산·감가상각(차량운반구)** = 물리적 차량 단위 → 재렌트해도 **동일 차량으로 유지**(최초/대 기준 키). 재렌트로 자산이 쪼개지면 안 됨.
- **수익(수납·청구·계약)** = 계약 인스턴스(재렌트별 번호) 단위.
